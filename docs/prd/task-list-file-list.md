# 任务列表内嵌文件列表 PRD（维护版）

> 状态：当前实现说明。最后按代码核对：2026-07-13。
>
> 本文是任务列表文件列表的产品与维护入口。历史设计稿、实施计划和报告仍保留用于追溯，但其中的“仅下载页”“每秒完整拉取”“待实施”等表述不是当前规范。

## 0. 维护决策原则

以普通用户的主路径为先：任务列表应稳定、直观，并且不能因展示功能额外改变下载行为。

- 修复高概率问题、后果严重的问题，以及改动小且不改变既有产品语义的问题。
- 对需要极端时序、断线或特殊多标签页操作才会触发的低频边界，先如实记录、观察真实反馈；不要为了假设场景引入复杂状态、额外常驻请求或牺牲主路径性能。
- 只有真实用户报告、使用方式变化，或能以很小且无副作用的改动消除风险时，才重新评估这些边界。

## 1. 目标与范围

用户无需逐个进入任务详情，就能在任务列表中查看每个任务的目录、文件大小和单文件进度。

功能适用于三个任务列表路由：

- `/downloading`（正在下载）；
- `/waiting`（等待/暂停）；
- `/stopped`（已完成、出错或已停止）。

它是一个共享、只读的展示模式：不改变 aria2 的下载、文件选择或任务状态，也不改变任务详情页的文件列表。

以下内容不在范围内：按文件名全局搜索、单个任务独立开关、分页/虚拟滚动、文件数量上限，以及在任务列表中修改文件选择。文件选择仍只能在任务详情页完成。

## 2. 用户体验

### 入口与持久化

- 三个任务列表页的顶部工具栏在 Help 后显示同一个“文件列表（Files）”开关；其他路由不显示，也不会清除选择。
- 开关默认关闭，使用一个持久化设置 `showFileListInTaskListPage` 控制三页。
- 升级兼容：首次读取新键时，如仅存在旧键 `showFileListInDownloadingPage`，沿用其布尔值并写入新键；新键始终优先。
- 任一页面切换开关会立即通知当前列表刷新；切换页面、刷新浏览器或重新打开应用后仍保持该选择。

### 展示与交互

- 仅当任务拥有非空 `files` 时，才在任务摘要下方显示内嵌面板；没有文件数据时不显示空面板或错误占位。
- 平铺任务显示文件名、文件进度条/百分比和文件大小；多目录 BitTorrent 任务显示目录树。
- 文件与目录复选框一律禁用，用于展示选择状态；目录展开/折叠仍可操作。
- 顶部搜索和任务排序以任务为单位：任务被隐藏或移动时，内嵌面板随它一起隐藏或移动。
- 平铺文件可按名称、进度、大小排序；多目录任务保持树形顺序，不能用文件排序破坏目录层级。
- 文件排序使用全局 `fileListDisplayOrder`，与任务详情页共用；在这里修改平铺任务排序，也会影响详情页。
- 文件区域阻止点击事件冒泡，不得误选外层任务、触发任务右键操作或进入详情页。

### 目录折叠状态

目录折叠状态以 `gid + nodePath` 隔离，父目录折叠会递归隐藏子树。它只保留在当前 `DownloadListController` 会话中：任务离开当前列表即清理，刷新页面、离开路由或重新打开浏览器后不恢复。

## 3. 数据与刷新契约

文件列表必须在“实时性、aria2 负载和浏览器 DOM 成本”之间保持以下平衡。

| 页面 | 首次、开启或结构失效时 | 稳定状态 | 新鲜度承诺 |
| --- | --- | --- | --- |
| `/downloading` | 请求完整任务字段 | 任务摘要按用户配置的刷新间隔走基础字段；文件明细最多每 5 秒请求一次完整字段 | 默认摘要约 1 秒；单文件进度约 5 秒加网络延迟 |
| `/waiting` | 请求完整任务字段 | 仅基础字段增量刷新；任务集合/结构变化后再完整请求 | 文件列表是最近一次完整读取的静态/准静态视图 |
| `/stopped` | 请求完整任务字段 | 仅基础字段增量刷新；任务集合/结构变化后再完整请求 | 文件列表是最近一次完整读取的静态视图 |

补充约束：

- 完整请求在基础字段之外包含 `dir`、`files`、`bittorrent` 与 `infoHash`；不对每个任务另发 `tellStatus`，避免 N+1 RPC。
- 下载页开启文件列表时，中间的基础响应只更新任务摘要，保留上一次完整响应中的文件树；每次完整响应才刷新文件 `completedLength` 与单文件进度。
- 若系统时钟回拨，下载页文件明细立即视为到期并请求完整字段，不能等待时钟追平。
- 若基础响应发现 GID/任务数量不再匹配，下一轮必须完整读取，不能把旧任务的文件树复用给新任务。
- 完整请求在飞行时阻止后续基础请求越过它；超过 30 秒仍无回包时解除该保护并允许重新读取。递增请求 ID 与模式 generation 会拒绝过期响应。
- 任务刷新间隔设为 `0` 时，控制器只做初始读取和用户触发的读取，不承诺实时更新。

## 4. 文件与目录数据模型

`aria2TaskService.processDownloadTask()` 负责把 aria2 原始任务转换为视图数据：

- 文件 `index`、`length`、`completedLength` 转为数字，派生 `fileName`、`selected` 与 `completePercent`。
- 长度为 `0` 的文件在任务完成时显示 `100%`，否则显示 `0%`，不得产生 `NaN`。
- 仅多目录 BitTorrent 任务构建虚拟目录节点；相对路径会去除下载目录和 torrent 根目录。
- 每次完整处理前必须删除旧虚拟节点，再从真实 `files` 重建；基础字段合并不得把虚拟节点当作真实文件重复处理。

这是维护时最重要的不变量：**完整响应重建文件树，基础响应保留文件树。**

## 5. 视觉与无障碍约束

- 内嵌面板是所属任务行的一部分，使用完整边框、左侧强调线、独立表头和文件行分隔来表达归属；不把整个任务列表改为卡片流。
- 浅色与深色主题均需覆盖面板、表头、行悬停和外层斑马纹，内层文件行不能继承造成混乱的外层背景。
- 只读复选框需要明显弱化，但目录展开图标必须保持可见、可点击。
- 窄屏下左右内缩缩小为 5px；文件名、进度和大小沿用现有 Bootstrap 列布局，不产生额外横向滚动。

## 7. 验收与回归重点

修改本功能时至少保持以下事实：

1. 三个任务列表页共用一个默认关闭、可迁移的开关。
2. 下载页不再每秒完整拉取所有 `files`；等待/停止页稳定后不周期性完整拉取。
3. 快速切换、乱序回包、丢失完整回包、时钟回拨、同长度 GID 替换都不会显示错误文件树或永久冻结刷新。
4. 目录树不重复，目录折叠按任务隔离，任务操作、搜索、排序、拖动和详情入口不回归。
5. 浅色、深色和窄屏下文件面板可读且不溢出。

先运行 `node test/task-list-file-list.test.js`，再按改动范围运行 `npm test`、`npx gulp lint` 与构建门禁。
