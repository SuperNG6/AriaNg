# BT 小文件过滤功能改进文档

> 状态：待评审  
> 日期：2026-07-14  
> 适用范围：本仓库新增的“新建 BT 任务自动过滤”和“过滤正在下载的 BT 任务”功能

## 1. 背景

当前项目有两条 BT 小文件过滤路径：

1. 新建磁力、远程 torrent 或本地 torrent 时，由自动过滤队列在 payload 下载前调整文件选择。
2. 在“正在下载”页面打开文件列表后，由批量过滤队列处理已经 active 的 BT payload 任务。

两条路径的执行状态机、持久化数据和恢复逻辑彼此独立，这是合理的；它们处理的任务来源、暂停语义和回滚方式不同，不应为了界面统一而合并。

当前问题主要出现在展示层和交互层：两套状态被独立呈现，但用户看到的是同一个“BT 小文件过滤”功能，因而会出现状态冲突、完成结果不可见、任务行与顶栏不一致，以及运行阈值与界面值不一致。

本轮改进以修正这些高概率、用户可见的问题为目标，不重写已经通过回归测试的执行状态机。

## 2. 改进原则

- 优先修复会误导用户、造成实际筛选结果与界面认知不一致的问题。
- 自动过滤和批量过滤继续使用现有独立队列及调度逻辑。
- 展示状态只从现有运行状态派生，不新增持久化队列或恢复协议。
- 优先使用已有翻译和状态字段；只有无法准确表达时才增加翻译键。
- 功能开发阶段只维护英文源文案和简体中文，其他语言在文案冻结后统一处理。
- 不为多标签页并发、极端回包乱序、浏览器崩溃窗口等低概率情况增加全局锁或复杂事务。
- 界面和跨功能问题优先连接浏览器验证；自动化测试只覆盖可能改错文件选择、任务状态或持久化结果的关键逻辑。
- 不以测试数量为目标，不为低概率分支、模板字符串或内部实现细节堆叠测试。

## 3. 本轮必须修复

### 3.1 顶栏过滤状态冲突

#### 当前问题

自动过滤使用 `btFileFilterStatus`，批量过滤使用 `bulkBtFileFilterStatus`。两者同时运行时，顶栏会渲染两个独立状态区域，例如：

```text
正在等待 1 个 BT 任务的文件列表
正在逐个过滤任务：0/1
```

两个状态没有来源说明，并且各自拥有 `aria-live`，视觉上和无障碍播报中都会产生冲突。

#### 目标行为

顶栏只渲染一个过滤状态容器。容器内部根据实际运行情况显示：

| 场景 | 展示行为 |
| --- | --- |
| 仅自动过滤运行 | 沿用当前自动过滤状态文案 |
| 仅批量过滤运行 | 沿用当前批量进度文案 |
| 两者同时运行 | 批量过滤状态优先；自动过滤不重复占用顶栏，但任务行徽标继续显示 |
| 批量完成、自动仍运行 | 批量完成提示优先保留约 5 秒，随后恢复自动状态 |
| 自动完成、批量仍运行 | 只显示批量运行状态，不叠加自动完成信息 |
| 两类都空闲 | 不显示状态容器 |

批量过滤是用户刚刚主动触发的操作，并且已有明确的总数进度，因此批量状态只要仍可见就优先。批量完成提示约 5 秒后隐藏；若自动过滤仍在运行，现有自动状态随后自然恢复。任务行徽标负责保留被暂时隐藏的自动任务可见性。

#### 实现边界

- 保留两个现有状态对象。
- 只用现有 `bulkBtFileFilterStatus.visible` 增加简单的显示优先级判断，不生成组合消息或第三个状态对象。
- 页面上只保留一个 `role="status"`、`aria-live="polite"` 容器。
- 批量操作栏可见时由操作栏承担批量状态展示，顶栏不重复显示同一批量状态。
- 不修改自动任务与批量任务的调度优先级。

### 3.2 批量过滤任务行缺少状态

#### 当前问题

任务行“过滤中”徽标只读取自动过滤队列。正在被批量过滤的 active BT 任务没有徽标，因此顶栏显示正在处理任务时，任务列表无法指出具体是哪一个任务。

#### 目标行为

- 自动过滤中的 root 和 child 继续显示现有徽标。
- 批量过滤当前正在处理的 GID 也显示“过滤中”。
- 批量任务阶段使用现有进度中的 `inspecting`、`applying`、`restoring` 派生，不新增持久化字段。
- `applying` 复用“正在应用过滤”，`restoring` 复用“正在恢复完整任务”。
- `inspecting` 需要准确表达“正在检查任务”，可增加一个阶段翻译键；不应错误显示为“等待文件列表”或“正在应用过滤”。
- 一个 GID 不会同时由自动队列和批量队列处理；现有 ownership 检查继续作为互斥保障。

徽标只表示过滤协调器正在处理该任务，不替代 aria2 自身的 active、waiting、paused 等任务状态。

### 3.3 运行阈值与界面值不一致

#### 当前问题

批量运行启动时会保存正确的阈值快照，但运行期间阈值输入框仍可修改。用户可能看到 `200 MB`，而当前批次仍按启动时的 `100 MB` 处理。

这会直接影响用户对实际文件选择结果的判断，必须修正。

#### 目标行为

- 批量运行继续使用启动时的不可变阈值快照。
- 批量状态对象从现有 definition 派生并公开本次运行阈值，不新增存储字段。
- 批量操作栏在运行期间明确显示“本次阈值”及其数值。
- “正在下载”页面的阈值输入框在批量运行期间禁用，避免用户误认为当前批次会实时变化。
- 用户在其他页面修改共享设置时，只影响后续新建任务和下一次批量运行；当前批次展示的快照不得随之变化。
- 批量完成并回到 idle 后，输入框恢复显示最新保存的共享设置。

不允许把运行中的 definition 阈值改写为新输入值，也不增加“动态调整当前批次”的能力。

### 3.4 完成提示被刷新提前清除

#### 当前问题

批量运行完成后，服务原本计划让完成提示显示约 5 秒；但文件列表打开时，新鲜任务详情响应会调用完成确认，并立即把状态重置为 idle。快速 RPC 环境中，界面可能从 running 直接跳到 idle，用户完全看不到完成结果。

文件列表关闭时提示反而能保留约 5 秒，导致行为依赖界面开关。

#### 目标行为

- 完成结果在文件列表打开或关闭时都保持约 5 秒。
- 新鲜完整响应只负责更新任务文件选择和防止旧预览重复提交，不负责提前隐藏完成提示。
- 完成提示计时结束后回到 idle。
- 如果完成提示期间启动了新的批量运行，新运行状态立即取代旧完成提示，旧计时器不得清除新状态。

#### 最小实现建议

- 保留现有 `bulkStatusVersion` 防止旧计时器覆盖新状态。
- 完成后的完整刷新继续执行，但 `acknowledgeBulkCompletion()` 不再直接调用 `setBulkIdleStatus()`。
- 由现有 5 秒计时器在版本仍匹配时统一进入 idle。
- 完成到新鲜完整响应之间，预览保持 analyzing 或不可提交；不引入新的持久化完成状态。

### 3.5 执行前文案与操作栏层级不合理

#### 当前问题

批量过滤操作栏在用户执行前长期展示以下内容：

- “仅处理正在下载的 BT 内容任务；不处理磁力元数据任务”；
- “若已选文件全部小于阈值，任务将保持不变”；
- “过滤 N 个任务”等容易被理解成隐藏或筛选任务列表的动作。

这些信息混合了功能标题、适用范围、例外规则和风险说明。执行前界面过重，真正需要用户确认的内容反而分散在菜单和弹窗两个位置。

#### 信息层级

操作栏只负责回答三个问题：

1. 这是“BT 小文件过滤”。
2. 当前有多少任务和小文件可以处理。
3. 本次使用什么阈值，以及从哪里开始执行。

适用范围、不会处理的任务、文件删除风险和无法安全过滤时的行为，统一放进点击“开始过滤”后的确认弹窗。顶栏全局状态只显示运行进度或结果，不展示执行规则。

#### 推荐中文文案

| 位置/状态 | 文案 |
| --- | --- |
| 操作栏标题 | `BT 小文件过滤` |
| 分析中 | `正在检查可过滤任务…` |
| 无候选 | `没有可过滤的任务` |
| 有候选 | `可处理 {{count}} 个任务，共 {{files}} 个小文件` |
| 阈值规则 | `排除小于 [100] MB 的已选文件` |
| 执行按钮 | `开始过滤` |
| 执行中按钮 | `过滤中`，保持禁用并显示 spinner |
| 执行进度 | `正在过滤：{{processed}}/{{total}}` |
| 完成结果 | `已完成：{{filtered}} 个任务已过滤，{{skipped}} 个未处理，{{failed}} 个失败` |
| 确认标题 | `确认过滤正在下载的 BT 任务？` |

确认正文集中说明：

```text
过滤 {{count}} 个任务，排除 {{files}} 个小于 {{threshold}} MB 的已选文件。
跳过磁力元数据、暂停及等待任务，保留手动排除。
失败时尝试恢复原选择；排除文件可能在任务完成时删除。
```

确认弹窗使用现有组件的纯文本换行能力，每条提醒在桌面端只占一行，不引入 HTML 或新弹窗组件；窄屏允许自然换行。

#### 界面行为

- 操作栏保持现有紧凑横条结构，不改成卡片、向导或独立页面。
- 执行前不常驻显示范围限制和删除警告。
- 候选摘要使用次要文本层级，阈值和按钮仍是主要操作。
- 输入错误继续在输入框附近显示，不移入确认弹窗。
- 运行期间使用阈值快照，禁用输入框和执行按钮；按钮不继续显示已经过期的候选任务数。
- 完成状态只显示结果统计，不重新展开规则说明。
- 优先修改现有翻译键的文案，不为每一句说明拆分新的状态或组件。

### 3.6 开发阶段不重复维护全部语言

#### 当前问题

当前维护规则要求每次新增或修改文案时，立即同步英文源文件和全部 10 个语言文件。功能和界面仍在调整时，这会反复翻译尚未冻结的文案，消耗大量时间，并容易在下一轮修改中全部返工。

#### 新的维护阶段

| 阶段 | 必须维护的语言 | 其他语言行为 |
| --- | --- | --- |
| 功能开发与调试 | `defaultLanguage.js` 英文、`zh_Hans.txt` 简体中文 | 新键回退英文；旧键允许暂时保留上一版翻译 |
| PR 合并前的一般回归 | 英文和简体中文键、占位符一致 | 不要求全部语言同步 |
| 文案冻结、准备发版 | 英文、简体中文及其余全部语言 | 一次性完成翻译并通过严格门禁 |
| 正式发版 | 全部语言 | 缺键或占位符错误时阻止发布 |

#### 开发门禁

日常 `npm test` 只检查：

- 新增或改动的英文源键存在；
- 简体中文存在对应键；
- 英文和简体中文的 `{{placeholder}}` 名称一致；
- 模板引用的键存在于英文源文件；
- 不出现空键、重复键或明显残留的临时文案。

日常开发不因为其他语言尚未更新而失败，也不要求为了通过测试机械复制英文到其他语言文件。

#### 发版门禁

准备发布版本时单独执行严格翻译检查：

1. 先冻结英文和简体中文文案，不再边翻译边修改产品表述。
2. 根据上一个发布版本到当前版本的文案差异，集中整理新增和变更的键。
3. 一次性更新其余语言文件。
4. 检查全部语言的键集合及 `{{count}}`、`{{processed}}` 等占位符。
5. 严格检查失败时阻止发布，但不阻塞前期功能开发。

严格检查使用独立命令 `npm run test:i18n-release`，并由 release workflow 调用，不再混入每次本地功能回归。实现本改进时，需要同步调整现有“全部语言必须始终相同”的测试和 `AGENTS.md` 维护规则。

## 4. 后续维护改进

以下问题需要记录，但不应阻塞本轮核心改进。

### 4.1 自动过滤首次入队持久化失败

当前自动过滤 `enqueue()` 不检查 `saveJobs()` 是否成功。浏览器正常可用时发生概率较低，本轮不扩展自动队列的持久化协议。

只有出现真实用户反馈，或后续能以局部改动完成时，才考虑让首次 `enqueue()` 返回失败并复用现有“过滤无法启动”提示。

后续状态写入失败、浏览器在 aria2 接收请求后立即崩溃、多标签页同时写入等低概率窗口继续作为已知限制，不实现事务日志或跨标签页锁。

### 4.2 维护文档与代码事实同步

完成代码改进后同步修正正式维护文档：

- BT 过滤专属翻译键实际为 35 个，不再写成 25 个。
- 代码已经使用 `lifecycleGeneration` 隔离旧生命周期回调，不再保留“没有 lifecycle epoch”的过时风险描述。
- 把“自动过滤”和“过滤正在下载任务”的共同展示契约写入同一份维护 PRD。
- 把多语言规则改为“开发阶段双语、发版阶段全量翻译”，并提供独立的发布检查命令。
- 历史计划只作为追溯资料，不作为当前行为的权威来源。

## 5. 明确不做

为控制维护成本，本轮不做以下工作：

- 不合并自动过滤队列和批量过滤队列。
- 不增加第三套统一持久化状态机。
- 不增加全局事件总线或通用通知中心。
- 不实现多标签页队列锁、跨浏览器同步或服务端协调器。
- 不为极端回包乱序建立事务、补偿日志或手工 GID 系统。
- 不重新设计 aria2 任务状态，也不改变 active、waiting、paused 的原有语义。
- 不顺带处理上游依赖漏洞、构建弃用警告或其他上游问题。
- 不进行与本次状态一致性无关的 UI 重构。
- 不在功能文案尚未冻结时反复翻译全部语言。

## 6. 验收标准

### 6.1 状态组合

1. 只有自动任务时，顶栏状态与现有行为一致。
2. 只有批量任务时，文件列表打开则操作栏显示进度；操作栏隐藏时顶栏显示进度。
3. 自动任务和批量任务同时运行时，批量状态优先，顶栏不同时显示两套进度。
4. 批量操作栏可见时，顶栏不重复显示批量状态；自动任务仍可通过任务行徽标识别。
5. 批量运行结束后，仍未完成的自动过滤状态重新显示。

### 6.2 任务行徽标

1. 自动 root/child 徽标行为保持不变。
2. 批量当前 GID 在 inspecting、applying、restoring 阶段显示徽标。
3. 当前批量项结算并切换到下一 GID 后，旧任务徽标消失，新任务徽标出现。
4. 批量运行完成、停止或切换 RPC 后不残留徽标。

### 6.3 阈值快照

1. 以 `100 MB` 启动后，即使共享设置变成 `200 MB`，当前 definition 仍为 `100 MB`。
2. 运行状态明确显示 `100 MB`，不能显示成新设置值。
3. 正在下载页面的输入框在运行期间不可编辑。
4. 下一次运行使用最新保存的设置。

### 6.4 完成提示

1. 文件列表打开时，完成提示不会被新鲜完整响应立即清除。
2. 文件列表关闭时，完成提示时长与打开时一致。
3. 完成结果约 5 秒后自动回到 idle。
4. 旧完成计时器不能覆盖随后启动的新批量运行。
5. 完成后的旧预览不能再次提交；新鲜完整响应到达后才允许下一次批量操作。

## 7. 验证策略

### 7.1 浏览器优先

本功能的主要风险是多套状态、路由、任务列表刷新和响应式界面之间的集成问题。开发过程中应先连接浏览器复现和验证，不以增加 Node 测试数量代替实际操作。

本轮必须在浏览器中检查：

1. 文件列表打开时，自动等待任务与批量任务同时运行，页面不显示冲突进度。
2. 运行中关闭文件列表、切换到其他页面，再返回“正在下载”，状态仍一致。
3. 执行前操作栏只显示标题、候选摘要、阈值和按钮；完整说明只在确认弹窗出现。
4. 运行阈值保持启动快照，其他页面修改设置不会改变当前批次展示。
5. 快速 RPC 响应下仍能看到约 5 秒的完成结果。
6. 批量当前任务显示徽标，切换到下一任务后徽标正确转移。
7. 375px 浅色和深色主题无溢出、遮挡或重复状态。

优先使用可控 aria2 测试任务；没有可用 aria2 时可使用浏览器 RPC mock，但最终发布前至少完成一次真实 aria2 主路径检查。

### 7.2 自动化测试聚焦功能和逻辑结果

自动化测试用于保护浏览器手工检查不容易稳定覆盖、且一旦出错可能影响真实任务的数据逻辑：

- 小于、等于和大于阈值时的 1-based 文件选择结果。
- 自动过滤与批量过滤不能同时修改同一个 GID。
- “立即下载”和“稍后下载”不能被错误暂停或启动。
- 批量运行使用不可变阈值快照。
- 完成计时器不能清除随后启动的新运行状态。
- 发生筛选失败时恢复原有选择和清理选项。

新增测试必须同时满足：

1. 输入和前置任务状态明确。
2. 断言的是文件选择、任务状态、RPC 参数、用户可见状态或持久化结果。
3. 测试失败意味着实际功能或业务逻辑错误，而不只是内部写法变化。
4. 在不改变产品行为的重构后，测试原则上仍然成立。

例如，“100 MB 阈值下生成 `select-file=2,3`”“稍后下载过滤完成后仍为 paused”“新鲜列表响应不能立即清除完成提示”是有效测试。仅断言某个函数名、依赖注入字符串、CSS 类或 HTML 片段存在，不足以证明功能正确。

同一行为原则上只保留一组能够证明最终结果的测试，不为每个内部函数、每种回调排列或每个模板分支分别建测试。

### 7.3 不再增加的低价值测试

- 只检查 HTML、CSS 或 JavaScript 是否包含某段字符串的测试。
- 与浏览器主路径重复、但不能验证最终界面结果的模板测试。
- 没有真实用户反馈的极端回包排列、浏览器崩溃时点和多标签页组合。
- 为追求覆盖率或测试数量而拆分出的重复用例。
- 大量模拟内部实现、重构后即失效但产品行为未变化的脆弱测试。

现有测试不在本轮大规模删除，避免把功能修复扩大成测试框架重构；修改到相关区域时，可顺手删除已经被浏览器验收或关键行为测试取代的低价值断言。

### 7.4 国际化检查

- 日常回归只检查英文源文件和简体中文的键及占位符。
- 新增键未同步到其他语言时，日常回归仍可通过并由运行时回退英文。
- `npm run test:i18n-release` 单独验证全部语言键集合和占位符完全一致。
- release workflow 未执行严格翻译检查时视为发布配置不完整。

## 8. 交付门禁

实现完成后必须通过：

```text
npm test
npx gulp lint
npx gulp clean build
npx gulp clean build-bundle
```

涉及界面时完成 375px 浅色和深色检查。测试或构建中出现的上游依赖告警单独记录，不扩大为本功能的修复范围。

准备正式发版时，在以上门禁之外再执行严格的全语言检查；该检查不进入日常开发门禁。

## 9. 推荐实施顺序

1. 先在浏览器中固定并发状态、完成提示和阈值错位的复现场景。
2. 只为文件选择、任务启停、阈值快照和完成计时器补最小失败回归。
3. 给现有批量状态增加当前 GID、阶段和阈值的派生字段。
4. 修正顶栏显示优先级，并扩展任务行阶段映射。
5. 整理操作栏文案，把范围和风险说明移入确认弹窗；开发阶段只更新英文和简体中文。
6. 修正完成确认与 5 秒提示生命周期。
7. 禁用运行中的批量阈值输入并展示本次快照。
8. 连接浏览器逐项回归，再运行现有测试、lint 和构建门禁。
9. 拆分日常双语检查和发版全语言检查，校准 `AGENTS.md` 与维护 PRD；自动入队持久化失败保留为观察项。
