# 主下载页文件列表功能 PRD

## 1. 文档信息

- 产品：AriaNg
- 功能名称：主下载页文件列表
- 生效页面：`/downloading`
- 文档日期：2026-07-11
- 状态：已确认，待实施

## 2. 背景

AriaNg 当前在主下载页以任务为单位展示名称、大小、总进度、剩余时间和下载速度。用户需要先进入某个任务的详情页，再切换到“文件列表”标签，才能查看该任务内每个文件的下载进度。

当多个多文件任务同时下载时，逐个进入详情页查看文件进度操作繁琐。用户希望在主下载页直接查看所有正在下载任务及其内部文件的实时进度，同时保留当前紧凑任务列表作为可切换的默认展示方式。

## 3. 目标

在 `/downloading` 页面顶部工具栏增加“文件列表”切换按钮。开启后，在每个任务行下方展开该任务的目录与文件列表，展示与现有任务详情“文件列表”一致的信息和交互；关闭后恢复现有任务列表。

功能需满足以下目标：

- 用户无需进入任务详情即可同时查看所有正在下载任务的单文件进度。
- 文件列表模式的目录层级、文件信息和进度表现与任务详情页保持一致。
- 用户的开关选择可以跨刷新和浏览器会话保存。
- 关闭文件列表模式后恢复现有的低带宽增量刷新机制。
- 不影响等待、已停止、任务详情和设置等其他页面。

## 4. 非目标

- 不在 `/waiting` 或 `/stopped` 页面提供该功能。
- 不修改任务详情页现有文件列表的产品行为。
- 不增加跨任务的文件搜索、文件筛选或文件级全局排序。
- 不提供单个任务独立展开或收起的开关；总开关开启后展示所有任务的文件列表。
- 不增加文件列表分页、虚拟滚动、按需加载或隐藏的文件数量上限。
- 不重构整个任务详情文件列表为通用组件。
- 不改变 aria2 的下载逻辑、任务状态和 RPC 刷新间隔配置。

## 5. 用户场景

1. 用户同时下载多个 BT 或 Metalink 多文件任务。
2. 用户进入“正在下载”页面，点击顶部“文件列表”。
3. 页面在每个任务下方展开对应文件，用户可以同时查看各文件进度。
4. 用户可展开或折叠多级目录，并沿用现有文件排序规则。
5. 用户再次点击“文件列表”，页面恢复当前紧凑任务列表。
6. 用户刷新或重新打开 AriaNg 后，页面恢复上次选择的模式。

## 6. 功能需求

### 6.1 功能入口

- 在顶部工具栏问号按钮右侧增加“文件列表”按钮。
- 按钮包含列表图标和文字“文件列表”。
- 中文界面使用现有 `Files` 翻译，显示为“文件列表”；其他语言复用对应的现有翻译。
- 按钮仅在当前路由为 `/downloading` 时显示。
- `/waiting`、`/stopped`、任务详情、设置及其他页面不显示按钮。

### 6.2 切换状态

- 首次使用时文件列表模式默认关闭。
- 关闭时沿用当前主下载任务列表。
- 开启时按钮使用工具栏现有蓝色激活色，提供明确的开启反馈。
- 点击已激活按钮可关闭文件列表模式。
- 开关值通过 AriaNg 设置服务写入本地存储；刷新页面、重新打开应用或离开后返回 `/downloading` 时恢复上次选择。
- 在其他页面隐藏按钮不应清除已保存的选择。

### 6.3 任务与文件布局

- 开启模式后保留现有任务行，不改变任务名称、总进度、剩余时间、速度和详情入口。
- 在每个任务行下方渲染该任务的文件列表。
- 文件区域必须处于所属任务的重复块内部，任务排序或过滤时随任务一起移动或隐藏。
- 相邻任务之间需有清晰的视觉分隔，避免文件归属不明确。
- 单文件任务显示一条文件记录。
- 没有取得 `files` 数据的任务只显示任务行，不渲染空白文件区域或错误占位。

### 6.4 文件列表行为

主下载页文件列表沿用任务详情页现有规则：

- 多目录任务以目录树展示。
- 目录支持展开和折叠。
- 文件行展示文件名、单文件进度条、进度百分比和文件大小。
- 沿用现有文件列表排序设置。
- 多目录任务沿用现有目录树顺序，不执行会破坏树结构的文件排序。
- 文件选择权限与详情页一致。由于本功能只在 `/downloading` 生效，活动任务的文件选择控件不可编辑。

目录折叠状态按任务隔离：

- 使用 `gid` 与目录 `nodePath` 共同标识折叠状态，避免不同任务的同名目录互相影响。
- 文件数据刷新后，仍然存在的目录保留当前页面会话中的折叠状态。
- 任务移出下载列表后清理其临时折叠状态。
- 目录折叠状态不跨页面刷新或浏览器会话持久化。

### 6.5 搜索与排序

- 顶部搜索继续以任务为过滤单位。
- 任务不匹配搜索条件时，其文件列表随任务一起隐藏。
- 本期不支持根据内部文件名匹配并保留任务。
- 主任务排序只调整任务顺序，文件列表始终跟随所属任务。
- 文件排序仅在每个任务内部生效，并共享现有文件排序设置。

### 6.6 原有操作兼容

文件列表模式不得改变以下现有行为：

- 任务选择和全选。
- 开始、暂停、删除、重试和清理任务。
- 任务右键菜单。
- 点击任务名称、文件数量或右箭头进入任务详情。
- 支持条件满足时的任务拖动排序。
- 顶部任务搜索和任务显示顺序设置。

文件区域内的点击事件不得冒泡为任务行选择操作。目录展开、文件行交互和任务选择应互不干扰。

## 7. 数据与刷新设计

### 7.1 关闭模式

- 完全沿用当前 `DownloadListController` 的刷新机制。
- 首次或结构失效时调用 `getTaskList('downloading', true, ...)` 获取完整字段。
- 后续调用基础字段请求并按 `gid` 增量合并。
- 不因用户曾经开启过文件列表而继续请求 `files`。

### 7.2 开启模式

- 切换开启后立即触发一次完整的正在下载任务刷新。
- 开启期间，每个刷新周期均通过一次 `tellActive` 请求获取完整任务字段，其中包含 `files`。
- 不为每个任务额外发起 `tellStatus`，避免请求数量随任务数增长。
- 每次响应同时更新任务总进度和各文件的 `completedLength`、`completePercent`。
- 处理任务数据时启用现有虚拟目录节点生成逻辑，使多目录任务得到与详情页一致的 `multiDir`、目录节点和层级信息。

### 7.3 关闭后的恢复

- 切换关闭后，后续刷新立即恢复基础字段请求。
- 已加载的文件对象可以暂留在当前内存任务对象中，但不得继续渲染或驱动完整字段请求。
- 普通任务刷新与现有结构完整性检查继续正常工作。

### 7.4 快速切换与响应时序

- 连续快速切换时，以最后一次保存的状态为准。
- 完整请求返回时若模式已经关闭，不得重新激活文件列表 UI。
- 基础请求返回时若模式已经开启，控制器必须保证随后进行完整刷新，不能长期显示旧的文件进度。
- 不允许因为切换而启动第二个独立的定时轮询器。

## 8. 状态设计

新增持久化布尔设置，语义为“在正在下载页面显示文件列表”：

- 默认值：`false`。
- 设置服务提供明确的 getter 和 setter。
- `MainController` 使用当前路由与该设置共同决定按钮是否显示及是否激活。
- `DownloadListController` 只在当前页面类型为 `downloading` 时读取该设置并改变刷新策略。
- 按钮切换通过共享状态或作用域事件通知当前下载列表立即刷新。
- `/waiting` 和 `/stopped` 虽共用 `list.html` 与 `DownloadListController`，也必须始终走原有列表逻辑。

## 9. 异常处理

- 开启功能后的首次完整刷新可使用现有全局加载状态。
- 定时刷新继续使用静默请求，不周期性遮挡任务列表或造成闪烁。
- RPC 请求失败时保留最后一次成功渲染的数据，并沿用项目现有连接错误处理。
- RPC 恢复后，在下一刷新周期自动更新文件进度。
- 单个任务缺少 `files` 时只显示任务行，不影响其他任务。
- 文件长度为 `0` 时沿用现有进度计算规则，不能产生 `NaN`、无效百分比或无效进度条宽度。
- 新任务在下一次完整刷新后自动出现文件列表。
- 完成或移出 `/downloading` 的任务按现有行为消失，其文件 DOM 与折叠状态同时清理。

## 10. 性能设计

- 开启模式会增加 `tellActive` 响应体和页面 DOM 节点数量，这是用户主动选择后的明确成本。
- 开启期间仍维持每刷新周期一次任务列表 RPC 请求。
- 不设置隐藏的任务数或文件数上限，避免出现部分任务缺少文件列表的歧义。
- 不增加单任务轮询、后台预加载或跨页面文件数据缓存。
- 关闭模式后必须恢复现有低带宽增量刷新，不能残留完整字段轮询。

## 11. 界面与响应式要求

- 按钮视觉需与现有顶部工具栏按钮一致。
- 激活状态需同时适配浅色和深色主题。
- 嵌套文件列表尽量复用现有 `.task-table`、进度条和文字样式，新增样式只负责缩进、归属分隔及点击区域。
- 桌面端按钮显示图标和文字。
- 窄屏下不得遮挡或覆盖搜索栏、任务列表或其他顶部操作；若顶部工具栏发生横向滚动，按钮应与现有工具项采用相同布局行为。
- 文件名较长时沿用现有换行或省略规则，不撑破进度和大小列。

## 12. 验收标准

### 12.1 入口与持久化

1. 进入 `/downloading` 时，问号右侧显示“文件列表”按钮。
2. 进入任意其他页面时不显示按钮。
3. 首次使用默认关闭。
4. 开启时按钮具有明确激活状态，再次点击可关闭。
5. 刷新、重新打开或离开后返回 `/downloading` 时保持上次选择。

### 12.2 文件展示

1. 开启后，每个有文件数据的下载任务下方显示所属文件列表。
2. 单文件任务显示一条文件记录。
3. 多文件任务显示全部文件。
4. 多目录任务显示与详情页一致的目录层级。
5. 不同任务的同名目录可以独立展开和折叠。
6. 文件名、大小、进度百分比和进度条显示正确。
7. 文件列表不会脱离所属任务或参与任务级排序。

### 12.3 实时刷新

1. 文件已下载长度变化时，主页面单文件进度按现有刷新间隔更新。
2. 新增任务后，其文件列表在下一次完整刷新时出现。
3. 任务完成并离开 `/downloading` 后，任务及文件列表同步消失。
4. 关闭模式后不再持续请求完整文件数据。
5. 再次开启时立即获取最新文件状态。

### 12.4 回归要求

1. 任务选择、全选、开始、暂停、删除、重试和右键菜单保持可用。
2. 目录或文件区域点击不会误切换任务选中状态。
3. 任务详情入口、搜索、任务排序和拖动排序规则不变。
4. `/waiting`、`/stopped` 和任务详情页行为不变。
5. 浅色、深色及窄屏布局均可正确使用。
6. 空任务列表和 RPC 断开状态不产生脚本错误。

## 13. 测试场景

- 无下载任务。
- 单个单文件 HTTP、FTP 或 SFTP 下载任务。
- 单个多文件 BT 任务。
- 包含多级目录的 BT 任务。
- 多个任务包含相同目录路径。
- 多个任务同时下载。
- 文件大小为零或尚未开始下载。
- 下载过程中新增、暂停、完成和删除任务。
- 快速连续开启和关闭文件列表模式。
- 开启后刷新浏览器及离开后重新进入页面。
- RPC 暂时断开后恢复。
- 切换任务搜索和不同任务排序方式。
- 浅色主题、深色主题及窄屏布局。

## 16. 方案决策

采用“主任务列表内嵌文件列表，开启期间使用完整 `tellActive` 刷新”的方案。

未采用以下方案：

- 为每个任务单独调用 `tellStatus`：请求数量随任务数增长，时序和错误处理更复杂。
- 抽取完整通用文件列表组件：会扩大到任务详情的选择文件、目录状态和右键菜单重构，超过本期必要范围。

该决策在实现复杂度、刷新准确性和现有架构兼容性之间取得平衡，并确保关闭功能后不改变 AriaNg 现有的低带宽行为。
