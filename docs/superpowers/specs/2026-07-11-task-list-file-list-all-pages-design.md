# 三类任务列表页文件列表扩展设计

## 1. 文档信息

- 产品：AriaNg
- 功能：任务列表页内嵌文件列表
- 生效页面：`/downloading`、`/waiting`、`/stopped`
- 文档日期：2026-07-11
- 状态：已确认，待实施

## 2. 背景

当前版本已在 `/downloading` 页面提供“文件列表”开关：开启后，每个任务下方展示该任务的目录、文件、大小和单文件进度，并记住用户选择。用户希望把相同行为扩展到“正在等待”和“已完成 / 已停止”页面，从三个任务列表页都能直接查看任务内文件，而不必逐个进入任务详情。

现有实现的设置名、控制器方法、事件名和 CSS class 均带有 `Downloading` 语义。如果只放宽路由判断，会留下误导性的公共接口，同时让等待和停止页面按每秒完整拉取 `files`，产生不必要的 RPC 响应和前端处理成本。因此本次采用通用任务列表模式，并按任务状态使用不同刷新策略。

## 3. 目标

- `/downloading`、`/waiting`、`/stopped` 顶部均显示同一个“文件列表”按钮。
- 一个持久化开关同时控制三个页面，并保留用户已有的下载页开关选择。
- 三个页面的文件区域在结构、排序、目录折叠、进度显示和视觉样式上保持一致。
- 三个页面的文件复选框均为只读；修改文件选择仍只在任务详情页完成。
- 下载中文件进度保持实时刷新；等待和停止页避免在每个轮询周期重复传输完整文件数组。
- 不改变任务操作、搜索、排序、拖动和详情页行为。

## 4. 非目标

- 不在任务详情、设置、服务器信息等其他页面显示该按钮。
- 不提供三个页面各自独立的开关。
- 不提供单任务独立展开开关、分页、虚拟滚动、文件名全局搜索或数量上限。
- 不允许在任务列表页修改 aria2 的文件选择状态。
- 不改变 aria2 RPC 刷新间隔。
- 不重写任务详情页文件列表组件。

## 5. 选定方案

采用“通用任务列表模式 + 状态感知刷新”。

- 把下载页专用的设置、控制器接口、事件和样式命名改为任务列表通用命名。
- 三个页面共享一个持久化布尔设置。
- `/downloading` 开启时每轮获取完整字段，以实时更新每个文件的进度。
- `/waiting` 和 `/stopped` 开启时只在首次进入、切换开启或任务结构失效时获取完整字段；其余轮询继续使用基础字段并合并到现有任务对象。
- 每次处理前先去除上一轮生成的虚拟目录节点，再根据真实 `files` 重建目录树，防止基础字段合并期间把虚拟节点误当真实文件重复处理。

未采用以下方案：

- 仅放宽 `/downloading` 路由判断：改动较小，但公共命名失真，三个页面都会持续拉取完整文件数组。
- 为每个页面保存独立开关：灵活度更高，但与用户确认的“一个选择控制三个页面”不一致，也增加了不必要的设置复杂度。

## 6. 入口与共享状态

### 6.1 工具栏按钮

- 问号按钮右侧继续使用现有列表图标和 `Files` 翻译文字。
- 当前路由为 `/downloading`、`/waiting` 或 `/stopped` 时显示。
- 三个页面中，按钮激活状态均来自同一个设置值。
- 切换页面不改变设置；刷新或重新打开浏览器后恢复上次选择。
- 任务详情和其他页面隐藏按钮，但不得清除设置。

### 6.2 设置命名与迁移

新设置键为 `showFileListInTaskListPage`，默认值为 `false`。设置服务公开：

- `getShowFileListInTaskListPage()`
- `setShowFileListInTaskListPage(value)`

设置服务必须在默认值合并之前检查原始持久化对象是否拥有对应属性，不能用已经补齐默认值的 `getOption()` 判断键是否存在。首次读取新键时执行一次兼容迁移：

1. 如果存储中已存在 `showFileListInTaskListPage`，直接使用。
2. 如果新键不存在但旧键 `showFileListInDownloadingPage` 存在，把旧布尔值写入新键并返回该值。
3. 如果两个键都不存在，写入并返回默认值 `false`。

旧键迁移后不再参与读取或写入，可留在历史本地设置中作为无效字段，避免为了清理字段引入额外存储变更。

## 7. 控制器与事件接口

下载页专用接口统一替换为通用任务列表命名：

- `MainController`
  - `isTaskListPage()`
  - `isTaskListFileListEnabled()`
  - `toggleTaskListFileList()`
- `DownloadListController`
  - `showTaskListFileList()`
  - `changeTaskListFileListDisplayOrder(type)`
  - `isSetTaskListFileListDisplayOrder(type)`
  - `getTaskListFileListOrderType()`
  - `isTaskListFileDirCollapsed(gid, nodePath)`
  - `collapseTaskListFileDir(gid, nodePath)`
- 作用域事件：`task-list-file-list-mode.changed`

事件只通知当前列表控制器立即按新模式刷新，不创建第二个定时器。控制器只在三个任务列表路由接受该事件；响应以当前设置为最终事实，事件参数仅用于触发刷新。

## 8. 数据流与刷新策略

### 8.1 通用处理流程

每次任务响应按以下顺序处理：

1. 使用现有响应序号机制拒绝过期响应，避免快速切换或重叠轮询回滚数据。
2. 完整响应直接使用真实任务数组；基础响应按 `gid` 合并到当前任务对象。
3. 对待渲染任务调用现有虚拟节点清理逻辑，移除上一轮生成的目录节点。
4. 当共享开关开启时，使用真实 `files` 重建 `multiDir`、目录虚拟节点、层级和进度派生字段。
5. 当共享开关关闭时，不生成虚拟目录节点，模板不渲染文件面板。
6. 清理已离开当前任务列表的目录折叠状态。

此顺序保证等待和停止页在基础字段刷新时保留已取得的真实文件数据，又不会累计重复的虚拟目录节点。

### 8.2 `/downloading`

- 开关关闭时沿用当前基础字段增量刷新。
- 开关开启后立即请求一次完整 `tellActive` 数据。
- 开启期间每个轮询周期请求完整字段，使文件 `completedLength`、百分比和进度条实时更新。
- 关闭后下一轮立即恢复基础字段刷新。

### 8.3 `/waiting`

- 开关开启或首次进入页面时请求一次完整等待任务列表，取得 `files`。
- 后续轮询使用基础字段并按 `gid` 合并。
- 现有完整性检查发现任务新增、删除、分页边界或结构变化时，再请求完整字段。
- 等待任务的文件进度随结构刷新更新；不承诺按每秒刷新静态或暂停状态下的文件数组。
- 任务拖动排序仍以外层任务行为单位，内嵌文件面板随任务整体移动。

### 8.4 `/stopped`

- 开关开启或首次进入页面时请求一次完整停止任务列表，取得 `files`。
- 后续轮询使用基础字段并按 `gid` 合并。
- 任务新增、清理、分页或结构变化触发完整字段刷新。
- 已完成 / 已停止任务的文件进度视为静态展示，不进行每轮完整文件刷新。

### 8.5 快速切换与失败恢复

- 开关快速切换时，以设置服务中最后保存的值为准。
- 开启后收到较早的基础响应时，必须继续安排完整刷新，不能长期显示旧文件数据。
- 关闭后收到较早的完整响应可以更新任务数据，但不能重新显示文件面板或恢复完整轮询。
- RPC 失败时保留最后一次成功数据，沿用现有连接错误处理；恢复后按当前路由和开关策略继续刷新。

## 9. 展示与交互

- 三个页面复用当前已精修的内嵌面板：完整边框、左侧蓝色归属线、独立表头、文件行分隔和浅色/深色主题覆盖。
- 样式 class 改为通用命名：
  - `has-task-list-file-list`
  - `task-list-file-list`
  - `task-list-file-panel`
- 所有新增及现有嵌套样式均以上述 class 为作用域，不影响任务详情页。
- 文件名、大小、单文件进度、目录树、目录展开/折叠和文件排序与当前功能一致。
- 三个页面的文件及目录复选框始终禁用；目录展开图标保持可点击。
- 点击目录或文件区域不得切换外层任务选择，也不得进入任务详情。
- 顶部搜索仍只匹配任务；任务被过滤时，其文件面板随任务一起隐藏。
- 无 `files` 或空 `files` 的任务只显示任务摘要，不显示空面板。

## 10. 性能边界

- 开关关闭时，三个页面都保持现有低带宽基础字段轮询。
- 只有 `/downloading` 在开关开启时每轮请求完整文件数据。
- `/waiting` 和 `/stopped` 仅在需要建立或修复文件结构时请求完整数据。
- 不新增按任务逐个调用 `tellStatus` 的 N+1 请求。
- 不设置隐藏文件数量上限；大任务的额外 DOM 成本由用户主动开启模式承担。

## 11. 预计修改模块

- `src/index.html`：把按钮显示及激活条件改为三类任务列表通用接口。
- `src/scripts/config/constants.js`：增加通用设置默认值。
- `src/scripts/services/ariaNgSettingService.js`：新增通用 getter/setter 和旧键迁移。
- `src/scripts/controllers/main.js`：通用路由判断、切换和事件广播。
- `src/scripts/controllers/list.js`：通用视图接口、状态感知刷新、虚拟节点规范化和响应时序保护。
- `src/views/list.html`：通用方法及 class 命名，保持只读文件选择。
- `src/styles/controls/task-table.css`、`src/styles/theme/default.css`、`src/styles/theme/default-dark.css`：把嵌套面板选择器改为通用命名。

## 12. 验收标准

1. 三个任务列表页均显示同一个“文件列表”按钮，其他页面不显示。
2. 任一页面切换后，另外两个页面立即采用同一状态；刷新和重开浏览器后保持。
3. 已保存旧下载页开关的用户升级后保留原选择。
4. 三个页面均可展示单文件、多文件及多级目录任务，样式和详情页信息一致。
5. 三个页面的文件复选框均不可编辑，目录可独立展开和折叠。
6. 下载页文件进度按现有轮询间隔更新。
7. 等待和停止页稳定后不在每轮 RPC 中请求完整 `files`。
8. 基础字段合并不会生成重复目录、重复文件或残留虚拟节点。
9. 任务搜索、选择、操作、右键菜单、详情入口、排序和等待页拖动不受影响。
10. 快速切换、页面切换、重叠响应和 RPC 断开恢复不产生模式回滚或脚本错误。
11. 浅色、深色及窄屏下保持当前内嵌面板的完整边缘和层级。

## 13. 验证方案

- 为设置迁移、共享开关、路由判断和刷新策略补充可执行测试；若现有工程缺少对应测试框架，至少用隔离的控制器/服务测试脚本覆盖关键分支。
- 执行项目 lint、标准版构建和 All-in-One 构建。
- 检查 RPC 请求字段：下载页开启时完整请求，等待/停止页稳定轮询时基础请求。
- 用模拟的完整响应、基础响应和乱序响应验证虚拟目录节点不会重复，最后一次开关状态不会回滚。
- 在真实 aria2 RPC 下手工验证三个页面的单文件、多文件、多目录、空列表、新增/删除任务和断线恢复。
- 分别检查浅色、深色、桌面和窄屏布局。
